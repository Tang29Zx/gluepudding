# 3D 实时站点大世界分层验证目标

来源：`REQUIREMENTS.md` 与 `TODO.md`

本文档定义每一层必须达到的最小验证目标。每一层都写清楚三件事：

- 进入本层前，上一层必须已经达到什么。
- 本层需要其他同学或外部系统准备什么。
- 本层最小验证目标是什么。

推进规则：

- 上一层没有通过，不进入下一层。
- 每一层都必须保留上一层能力。
- 如果新增能力破坏上一层能力，先修复回归再继续。
- 依赖外部服务的层，必须先有模拟版本。
- 每一层的问题、修复和验证过程记录在该层目录的 `debug.md`，路径为 `validation/layer-N/debug.md`。
- 真实设备控制只能在权限和安全边界明确后接入。
- 涉及凭据、token、设备密钥时，只记录配置位置，不记录值。

## Layer 0：静态兜底层

目标：即使 3D 不可用，站点仍然可进入、可理解、可导航。

进入本层前必须满足：

- 无上一层要求。
- 已确认 `gluepudding` 是总站入口。
- 已确认核心体验最终要做成 3D 实时大世界，而不是跳转式导航。

需要其他同学或外部系统准备：

- 暂不依赖其他同学。
- 暂不依赖后端、RDK、WebRTC、IoT 或五子棋真实能力。

本层最小验证目标：

- `gluepudding` 首页能打开。
- WebGL 不可用时显示 2D 兜底内容。
- 兜底内容包含占卜屋、实验室、五子棋三个入口说明。
- 兜底内容不暴露任何 token、密钥、设备地址或内部错误。
- 页面在桌面端和移动端都没有文字重叠。

通过标准：

- 禁用 WebGL 或模拟 3D 初始化失败时，用户不会看到空白页。
- 用户能看懂这是一个 3D 大世界入口，并知道哪些能力暂不可用。

## Layer 1：3D 世界存活层

目标：先证明大世界本身能渲染和运行。

进入本层前必须满足：

- Layer 0 已通过。
- 静态兜底页可用。
- 3D 初始化失败时能回落到静态兜底。

需要其他同学或外部系统准备：

- 暂不需要建模同学交付正式 GLB。
- 暂不需要后端接口。
- 暂不需要真实 WebRTC、RDK 或 IoT 服务。

本层最小验证目标：

- 进入 `gluepudding` 首页后显示 3D 场景。
- 场景不是空白画布。
- 地面、灯光、相机、玩家占位体可见。
- 没有外部服务时，世界仍然能加载。
- 3D 初始化错误能被捕获，并回落到 Layer 0。

通过标准：

- 刷新页面 3 次，3D 场景都能稳定出现。
- 浏览器控制台没有阻断运行的未捕获错误。

## Layer 2：移动和相机层

目标：证明用户可以在世界中行动。

进入本层前必须满足：

- Layer 1 已通过。
- 3D 世界能稳定渲染。
- 兜底层仍然可用。

需要其他同学或外部系统准备：

- 暂不需要正式场景模型。
- 暂不需要真实业务接口。
- 如果建模同学有初版空间尺寸建议，可以用于出生点和区域距离参考，但不是阻塞项。

本层最小验证目标：

- `WASD` 可以移动玩家。
- `Shift` 可以让玩家疾跑。
- `Space` 可以让玩家跳跃并落回地面。
- 相机能跟随或稳定观察玩家。
- 玩家移动不依赖帧率。
- 玩家不会一开始就卡进模型或掉出地面。
- `ESC` 不会导致页面异常。

通过标准：

- 用户可以从出生点移动到占卜屋、实验室、棋盘三个区域附近。
- 移动过程中场景不闪烁、不黑屏、不丢失相机目标。

## Layer 3：世界内交互层

目标：证明大世界不是静态展示，而是可交互空间。

进入本层前必须满足：

- Layer 2 已通过。
- 玩家能稳定移动到三个区域附近。
- 相机和移动不会阻塞交互检测。

需要其他同学或外部系统准备：

- 暂不需要真实后端。
- 暂不需要正式模型。
- 建模同学如果能提供交互点命名建议，可以后续映射；本层先用代码固定坐标。

本层最小验证目标：

- 交互对象有统一配置。
- 靠近占卜屋显示提示。
- 靠近实验室显示提示。
- 靠近棋盘显示提示。
- `E` 可以触发区域级交互。
- 准星命中对象后，左键可以触发对象级选择。
- 被选中的对象有可见高亮。
- 打开面板时暂停移动，关闭后恢复移动。

通过标准：

- 用户能在不跳转页面的情况下，分别触发三个区域的世界内交互。
- 左键选中一个对象后，界面能明确显示“当前选中了什么”。

## Layer 4：模块外壳层

目标：证明占卜屋、实验室、五子棋都能作为常驻 3D 世界模块表面直接操作，而不是整页跳转或打开 DOM 浮层。

进入本层前必须满足：

- Layer 3 已通过。
- 三个区域都能触发交互。
- 玩家移动、准星命中和左键选择不会互相阻塞。

需要其他同学或外部系统准备：

- 暂不需要真实业务数据。
- 暂不需要真实推流或设备。
- 如果其他同学已有接口草案，可以开始对齐字段，但本层不阻塞。

本层最小验证目标：

- 占卜屋模块常驻贴在占卜屋入口表面。
- 实验室模块常驻贴在实验室大屏表面。
- 五子棋模块常驻贴在棋盘附近的 3D 表面。
- 每个模块都有加载中、错误、离线三类状态。
- 准星命中模块状态控件后，左键或 `E` 能在 Canvas 内切换状态。
- 任一模块报错不会导致整个世界崩溃。
- 模块操作不会主动退出 Pointer Lock，也不会暂停玩家移动。

通过标准：

- 连续操作三个模块的状态控件，玩家仍然停留在原世界中。
- 模块表面操作时鼠标焦点保持在 3D Canvas。
- 地址栏不发生整页跳转。

- Layer 4 已通过。
- 常驻模块表面、准星和 Canvas 内交互口径已确认。

## Layer 5.5：视觉渲染基线层

目标：在继续接入实验室、五子棋和占卜交互前，先把大世界的基础光照、屏幕材质和低成本高画质感统一到可持续的视觉基线。

进入本层前必须满足：

- Layer 5 已通过。
- 占卜屋模型摆放已经验收，不能在本层重新调整模型坐标。
- 世界内移动、准星、实验室模块表面和五子棋模块表面仍可用。

需要其他同学或外部系统准备：

- 不需要后端、WebRTC、RDK、IoT 或五子棋真实能力。
- 不需要新增模型资源或后处理依赖。
- 如建模同学后续提供烘焙贴图、HDRI 或压缩模型，可进入 Layer 13 或后续美术专项再接入。

本层最小验证目标：

- 3D renderer 使用 sRGB 输出、ACES tone mapping 和软阴影配置。
- 世界光照从高环境光改为低环境光、暖色主光和柔和天光。
- 主光阴影范围覆盖出生点、占卜屋、实验室和五子棋附近，阴影不出现大面积闪烁。
- 占卜屋三块内容屏从白色基础面片改为带轻微发光的深色屏幕材质。
- 实验室和五子棋常驻模块面板具有屏幕质感，按钮和文字仍然可读。
- 占卜屋内部有低强度暖光、紫色辅助光和星座侧弱冷光。
- 不新增 Bloom、SSAO、WebGPU 或真实 path tracing。
- 不实现占卜、实验室或五子棋业务交互。

通过标准：

- `npm run assets:fortune:check`、`npm run build` 和 `npm run assets:check` 通过。
- 首页能进入 3D 世界，不进入 2D fallback。
- 桌面端和移动端截图没有黑屏、过曝、明显文字重叠或 HUD / 触屏按钮遮挡主要内容。
