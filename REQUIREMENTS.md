# 3D 实时站点大世界需求文档

来源：用户当前设想与 `fortune_world_complete_plan.md`

本文档是当前规划事实来源。旧方案里的占卜内容继续保留为“占卜屋”模块输入，不再把项目理解成单独的算命页面，也不再把 IoT、五子棋当成跳转链接。

## 1. 我的目标

把 `gluepudding` 总站导航升级为一个实时 3D 大世界。

用户进入总站后，不是看到普通卡片导航，也不是跳转到其他站点，而是在同一个 3D 世界中移动、观察和操作：

- 进入占卜屋，直接在 3D 桌面上选塔罗牌、做星座和周易占卜。
- 进入实验室，在大屏幕上实时观看 WebRTC 推流。
- 在实验室中实时渲染 RDK 构建的世界模型。
- 在实验室控制台里直接控制 IoT 设备，例如开门。
- 在草地或棋盘区席地而坐，直接打开并游玩五子棋。

核心目标是：`gluepudding` 变成一个统一的 3D 交互外壳，现有 `iot`、`game`、`auth` 等项目不再只是链接目标，而是被接入为世界里的实时能力。

## 2. 核心原则

### 2.1 不跳转原则

核心交互不通过整页跳转完成。

必须避免这种体验：

```text
靠近实验室 -> 跳转到 iot.gluepudding.com
靠近棋盘 -> 跳转到 game.gluepudding.com/wuziqi/
靠近占卜屋 -> 跳转到另一个 /world 页面
```

目标体验应该是：

```text
靠近实验室 -> 实验室大屏亮起，直接显示实时画面和控制台
靠近棋盘 -> 世界内出现棋盘交互或嵌入式棋局面板
靠近占卜屋 -> 世界内打开占卜桌、卡牌、铜钱和结果面板
```

认证、权限不足、设备不可用等异常可以显示世界内弹窗或面板；不应默认把用户带离大世界。

### 2.2 现有项目作为能力来源

现有项目边界保留，但对用户表现为同一个世界：

```text
gluepudding/  -> 3D 世界外壳和统一体验入口
iot/          -> IoT 能力来源：门禁、设备状态、WebRTC/控制接口
game/         -> 游戏能力来源：五子棋规则、AI、资源
auth/         -> 认证和权限能力来源
```

### 2.3 实时优先

世界里的关键对象应该承载实时状态：

- 实验室屏幕显示实时视频流，不是静态图片。
- RDK 世界模型可实时更新或周期刷新，不是只放一张截图。
- 门禁按钮显示当前状态，并能发送真实控制命令。
- 占卜卡牌由 3D 选中行为触发，不只是表单按钮。
- 五子棋在世界内可交互，不只是入口链接。

### 2.4 中文用户优先

本项目明确面向中文用户。

- README、需求文档、技术说明、验证记录和协作文档默认使用中文。
- 产品 UI 文案默认使用中文，避免中英文混杂影响普通用户理解。
- 代码标识、文件名、类型名、接口路径、命令、环境变量和第三方协议名保持英文。
- 如果接入外部英文库或协议，面向用户的说明需要转成中文表达。

## 3. 我的交付范围

我负责 3D 大世界主框架、交互系统和现有能力接入：

- 将 `gluepudding` 总站升级为 3D 大世界入口。
- 搭建 3D 场景、玩家移动、相机、交互检测和世界内提示。
- 建立世界内模块系统，让占卜屋、实验室、棋盘作为场景区域运行。
- 实现占卜屋前端体验：星座、塔罗、周易。
- 实现塔罗卡牌的 3D 选择交互，例如准星射线命中后左键选中某张牌。
- 实现实验室大屏：可把 WebRTC 视频流作为纹理挂到 3D 屏幕上。
- 实现 RDK 世界模型的前端接入和实时渲染容器。
- 实现实验室门禁控制面板，调用 IoT 门禁能力。
- 实现五子棋在世界内打开和游玩，不使用整页跳转。
- 提供模拟数据和模拟设备状态，保证外部服务不可用时仍可演示。
- 提供 WebGL 失败或性能不足时的 2D 兜底入口。

## 4. 不属于我的交付范围

以下内容不是我的实现交付物，只作为外部能力或接口输入：

- 不负责制作最终美术模型，但需要能加载已交付 GLB。
- 不负责实现 WebRTC 推流服务端，但需要能消费推流。
- 不负责实现 RDK 建图或仿真算法，但需要能渲染其输出。
- 不负责实现门锁硬件和 MQTT 桥接，但需要能调用控制接口。
- 不负责重写认证系统，但需要能识别登录态和权限状态。
- 不负责从零重写五子棋 AI；优先复用现有五子棋能力。

## 5. 目标世界结构

建议世界结构：

```text
中心出生点
├── 占卜屋
│   ├── 星座台
│   ├── 塔罗桌
│   └── 周易桌
├── 实验室
│   ├── WebRTC 大屏
│   ├── RDK 世界模型展示台
│   ├── 门禁控制台
│   └── 设备状态墙
└── 草地棋盘
    └── 五子棋棋局
```

第一版可以使用占位几何体：

- 占卜屋：小屋或桌面区域。
- 实验室：房间、屏幕、控制台。
- RDK 展示台：透明地台或悬浮模型台。
- 五子棋区：地面棋盘或坐垫棋桌。

## 6. 主要交互

### 6.1 玩家移动

- `W`：向前移动。
- `S`：向后移动。
- `A`：向左移动。
- `D`：向右移动。
- `Shift`：按住疾跑。
- `Space`：跳跃。
- 鼠标控制视角或目标方向。
- `E`：区域级主交互，例如进入区域或打开区域面板。
- 左键：对象级主交互，例如选中塔罗牌、操作屏幕、点选模型对象。
- `ESC`：关闭当前世界内面板。

移动和交互必须支持暂停：当用户正在操作占卜桌、实验室控制台或棋局时，玩家移动可以暂停，避免误操作。

### 6.2 世界内交互方式

交互需要以 3D 射线检测为基础：

- 玩家靠近区域时显示提示。
- 鼠标悬停对象时显示高亮。
- `E` 用于进入区域级交互。
- 准星对准对象后，左键用于对象级选择或上下文操作。
- 移动端使用点击或屏幕按钮替代左键。
- 选中对象后，世界内面板展示详情或操作按钮。

典型例子：

```text
准星对准塔罗牌后左键 -> 该牌高亮并加入已选牌
准星对准实验室大屏后左键 -> 放大查看 WebRTC 画面
准星对准 RDK 模型节点后左键 -> 查看节点状态
准星对准门禁控制台后左键 -> 打开开门/状态面板
准星对准棋盘交叉点后左键 -> 落子
```

## 7. 占卜屋需求

占卜屋是世界内的一个区域，不是独立跳转页。

### 7.1 星座台

- 用户靠近星座台后可打开星座面板。
- 输入生日或选择星座。
- 调用真实接口或模拟数据。
- 在世界内展示星座名、日期、综合分、爱情分、事业分、健康分、幸运色、幸运数字、宜、忌和总结。

### 7.2 塔罗桌

- 塔罗牌以 3D 对象展示在桌面或环形牌阵中。
- 用户可以准星对准并左键选中卡牌。
- `single` 模式选择 1 张。
- `three_card` 模式选择 3 张。
- 选中卡牌需要高亮。
- 返回结果后卡牌翻面。
- 展示牌名、正逆位、关键词、解释和总结。

### 7.3 周易桌

- 用户在世界内打开周易面板。
- 输入问题后点击起卦。
- 铜钱可以用 3D 动画或第一版占位动画表示。
- 结果展示六爻、本卦、变卦、变爻和总结。
- 后端返回六爻顺序是从下往上，前端展示时必须反转：

```ts
result.lines.slice().reverse()
```

## 8. 实验室需求

实验室是世界内实时设备和机器人能力区域。

### 8.1 WebRTC 大屏

实验室里必须有一块 3D 大屏。

目标体验：

- 用户走进实验室即可看到大屏。
- 有推流时，大屏显示实时 WebRTC 画面。
- 用户准星对准大屏并左键点击可放大查看。
- 推流断开时，大屏显示离线状态，而不是破坏整个世界。

前端要求：

- 支持把 `MediaStream` 绑定到隐藏 `video` 元素。
- 支持把 `video` 作为 Three.js 视频纹理贴到屏幕模型上。
- 支持连接中、在线、断线、无权限四种状态。
- WebRTC 信令地址和流标识通过环境变量或配置传入，不硬编码密钥。

### 8.2 RDK 世界模型展示

实验室中需要一个 RDK 世界模型展示台。

目标体验：

- RDK 构建的世界模型可以实时出现在大世界的实验室里。
- 用户可以旋转、缩放、查看模型。
- 如果模型数据持续更新，展示台同步刷新。
- 如果实时数据不可用，展示占位模型或最近一次可用模型。

可接受的数据来源：

- GLB / glTF 模型文件。
- WebSocket 推送的场景状态。
- 周期拉取的模型元数据。
- 后续约定的 RDK 场景描述格式。

第一版要求：

- 先支持加载静态 GLB 或模拟模型。
- 预留实时更新接口。
- 不把 RDK 数据格式写死在渲染组件里，使用适配层转换。

### 8.3 门禁控制台

实验室里需要一个门禁控制台。

目标体验：

- 用户在实验室里打开门禁控制台。
- 控制台显示当前门禁状态。
- 用户确认后可以发送开门指令。
- 成功、失败、无权限、设备离线都要在世界内反馈。

安全要求：

- 不能在前端硬编码 token、密码、私钥或设备密钥。
- 控制门禁前必须依赖 `auth` 的登录态和角色权限。
- 开门操作必须有明确用户确认。
- 门禁控制接口失败不能影响 3D 世界运行。
- 日志和错误提示不能泄露敏感信息。

## 9. 五子棋需求

五子棋也必须在大世界内打开，不使用整页跳转。

可分两阶段：

### 9.1 第一阶段：世界内嵌入式棋局

- 用户走到草地棋盘区域。
- 按 `E` 或准星对准棋盘后左键点击，世界内打开棋局面板。
- 复用现有五子棋页面或逻辑。
- 棋局面板表现为世界内浮层、棋桌屏幕或嵌入式界面。
- 用户可以直接落子和对弈。
- 关闭棋局后回到大世界。

### 9.2 第二阶段：原生 3D 棋盘

- 棋盘成为 3D 场景对象。
- 用户准星对准棋盘交叉点后左键落子。
- 棋子以 3D 对象显示。
- AI 结果直接反映到 3D 棋盘。
- 现有五子棋 AI 和规则逻辑通过适配层复用。

## 10. 接口和能力契约

### 10.1 占卜接口

占卜接口继续沿用统一格式：

- `GET /api/fortune/health`
- `POST /api/fortune/zodiac`
- `POST /api/fortune/tarot`
- `POST /api/fortune/iching`

成功返回：

```json
{
  "module": "zodiac",
  "success": true,
  "data": {}
}
```

失败返回：

```json
{
  "module": "zodiac",
  "success": false,
  "error": "birthday is required"
}
```

### 10.2 IoT 能力接口

前端需要通过适配层调用 IoT 能力，不能把设备调用散落在场景组件里。

第一版至少需要：

- 查询门禁状态。
- 发送开门指令。
- 查询设备在线状态。

具体接口路径以现有 `iot` 后端或后续网关为准。前端需要把这些能力封装成类似：

```ts
getDoorState()
openDoor()
getDeviceStatus()
```

### 10.3 WebRTC 能力接口

前端需要通过配置获取：

- 信令服务地址。
- 流标识。
- ICE 配置来源。
- 权限失败时的错误码。

不得在代码或文档中保存真实凭据。

### 10.4 RDK 模型能力接口

前端需要通过适配层获取 RDK 世界模型：

```ts
getRdkWorldSnapshot()
subscribeRdkWorldUpdates()
```

适配层负责把外部数据转换成渲染层可消费的数据，渲染组件只关心模型、节点、状态和更新时间。

### 10.5 五子棋能力接口

第一阶段可以内嵌现有五子棋页面或抽离现有五子棋逻辑。

如果使用嵌入式页面，需要解决：

- 同源或跨源嵌入限制。
- 键盘事件和大世界控制冲突。
- 面板关闭时状态保留或重置策略。

如果抽离逻辑，需要把规则、AI、棋盘状态封装成适配层。

## 12. MVP 标准

第一版最小可行版本必须证明“同一个 3D 世界内实时打开能力”：

- `gluepudding` 首页显示 3D 大世界。
- 玩家可以移动。
- 世界中有占卜屋、实验室、棋盘三个区域。
- 靠近区域显示提示。
- 占卜屋内可以完成至少塔罗选牌流程。
- 塔罗至少支持准星对准后左键选择卡牌并显示结果。
- 实验室有 3D 大屏，可显示模拟视频或真实 WebRTC 流。
- 实验室有门禁控制台，可用模拟状态演示；真实接口可用时能调用真实控制。
- RDK 展示台能加载一个静态或模拟模型，并预留实时更新入口。
- 五子棋能在世界内打开，不整页跳转。
- 外部服务不可用时，世界仍然可运行并展示离线状态。

## 13. 理想完成标准

理想版本包括：

- WebRTC 推流稳定贴到实验室大屏。
- RDK 世界模型能实时更新。
- 门禁控制走真实权限校验和真实设备接口。
- 五子棋为原生 3D 棋盘，左键或点击可落子。
- 占卜屋有完整星座、塔罗、周易三个互动桌。
- 所有能力都在大世界内打开，不出现整页跳转。
- 低性能设备可以切换 2D 兜底模式。
- 移动端有可用的触屏控制方案。

## 14. 风险和约束

- WebRTC、门禁控制、RDK 实时模型都依赖外部服务，必须有模拟模式。
- IoT 控制属于安全敏感能力，必须走认证和权限检查。
- 视频纹理和复杂模型可能影响性能，需要按需加载和降级策略。
- 五子棋现有页面如果跨源嵌入受限，需要通过同源代理、代码抽离或模块化改造解决。
- 右键不作为主交互；移动端必须提供触屏替代操作。
- 任何凭据、token、设备密钥都不能写入前端代码、文档或日志。
